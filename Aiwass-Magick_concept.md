## **1\. コンセプト概要**

### **1.1 背景と目的**

「Aiwass-Magick」は、LLM（Large Language Model）を活用した自律・動的な学習フレームワークである。

既存の学習ツールやRAG（Retrieval-Augmented Generation）が提供する「AIが生成した静的なレポートを読むだけ」の退屈な体験を排し、「ユーザーの主体的意志（Will）に基づき、コンテキストの軸（重力）を維持したまま、Wikipediaのリンクを無限に踏み続ける（穴に落ちる）知的興奮」をシステムとして再現・設計する。

### **1.2 コア体験 (Thelema Engine)**

* **意志の重力 (Will Anchor):** 「GCP ACE試験合格」などの最終目標（軸）をプロンプトとして固定。無限拡散（ブレ）を防ぎつつ、関連領域をサジェストする。  
* **魔術的インターフェース (The Magick UI):** WebUI上のボタン（用語チップ、関連トピック）をポチポチと押すだけで、全可能性の空間（Nuit）から今欲する知識（Hadit）を動的にプロビジョニング（物質化）する。  
* **顕現のログ (Obsidian Sync):** 巡礼（探索）した知識の履歴は、自動的にローカルの知識ベース（Obsidian）にMarkdownファイルとして自動配置される。

## **2\. システムアーキテクチャ**

## **3\. コア・データ構造 (API レスポンス JSON 規格)**

Aiwass-Magickの挙動の全ては、バックエンドがLLMから生成・安定出力させる以下の「構造化JSON（Structured Output）」に依存する。

{  
  "breadcrumb": \[  
    "GCP基礎",  
    "ネットワーク",  
    "VPC"  
  \],  
  "explanation": "VPCはプロジェクト単位のグローバルL3ネットワークで、グローバルリソースとして定義されます。サブネットはリージョンごとに作成され、内部IPのCIDR範囲を割り当てます...",  
  "term\_suggestions": \[  
    {  
      "term": "CIDR",  
      "hint": "IPアドレス範囲をスラッシュ（/）表記で表す方法"  
    },  
    {  
      "term": "ルートテーブル",  
      "hint": "パケットが宛先に到達するための経路ルール"  
    }  
  \],  
  "related\_topics": \[  
    {  
      "topic": "Cloud NAT",  
      "reason": "外部IPを持たないインスタンスの安全なアウトバウンド通信に必要"  
    },  
    {  
      "topic": "VPC Peering",  
      "reason": "別VPC間を内部IPのままセキュアに接続する手段"  
    }  
  \],  
  "magick\_metadata": {  
    "current\_will\_vector": "gcp\_ace\_network\_focus",  
    "obsidian\_path": "GCP/Network/VPC.md"  
  }  
}

### **パラメータ定義**

1. **breadcrumb (配列):** 現在の探索階層の視覚化。  
2. **explanation (文字列):** メインとなる知識解説。Markdownフォーマットを許容。  
3. **term\_suggestions (配列オブジェクト):** 文脈内で登場したコア専門用語。クリックするとその用語の解説をオーバーレイまたは即時取得。  
4. **related\_topics (配列オブジェクト):** 次の「巡礼先」候補。クリックするとそのトピックを主軸とした新しい解説画面へダイブする。  
5. **magick\_metadata (オブジェクト):**  
   * current\_will\_vector: 初期意志（コンテキストの軸）への吸着強度、または現在の追従ベクトル名。  
   * obsidian\_path: Obsidianに書き出す際のVault内の推奨保存パス。

## **4\. エージェント設計 (4つの役割)**

バックエンドのOrchestratorは、必要に応じて以下の4つの自律エージェントにプロンプトをディスパッチする。

| ロール (Role) | 役割 (Task) | 備考 |
| :---- | :---- | :---- |
| **Planner (プランナー)** | ユーザーの意志（初期トピック）からロードマップと探索インデックスを初期構築する。 | システムの「骨組み」のプロビジョニング |
| **Researcher (リサーチャー)** | Web検索（Google Search API）や既存PDF等の素材を漁り、生情報を収集・集約する。 | 外部ナレッジの動的インポート |
| **Synthesizer (シンセサイザー)** | 収集データを整理・構造化し、規格化された「魔法のJSON」およびMarkdownを錬成する。 | 最重要エンジン（Claude API等の高度LLMに担わせる） |
| **Quizzer (クィザー)** | 生成した知識ログから、記憶定着のための「確認テスト」や「フラッシュカード」を自動生成する。 | 試験勉強（GCP ACE等）向けの自動評価レイヤー |

## **5\. 実装フェーズ**

### **【Phase 1】 召喚陣の構築とスタティック疎通（プロトタイプ）**

* **FastAPIエンドポイント (/dive) の作成:**  
  上記の「魔法のJSON」をスタティックに返すダミーAPIの実装。  
* **React \+ Vite \+ Tailwind CSS フロント構築:**  
  レスポンスJSONを受け取り、パンくず、説明エリア、用語チップ、関連トピックカードを美しく表示するUI（魔方陣インターフェース）の作成。クリック時に次のスタティックデータをAPIにリクエストする疎通テスト。

### **【Phase 2】 意志（Will）の結合（LLM API連携）**

* **System Promptの錬成:**  
  Claude APIを用いて、どんなインプットに対しても狂わずに「魔法のJSON規格」を100%出力させるプロンプトエンジニアリング（Pydantic等を用いたStructured Outputsの保証）。  
* **履歴スタック（コンテキスト継続）の実装:**  
  ユーザーがポチポチした履歴（巡礼ログ）をFastAPI側でセッションまたはインメモリスタックに格納し、「ブレずに深掘りする」コンテキスト制御を実装。

### **【Phase 3】 顕現の固定化（Obsidian同期 ＆ ローカル保存）**

* **FastAPIからのMarkdown自動排出:**  
  /dive で決定した各ノードを、しなりのObsidian Vault（または指定ディレクトリ）へMarkdownファイルとして自動出力。  
* **フロントに「Obsidianに顕現（保存）完了」ステータスを表示。**

### **【Phase 4】 自律探索機能の拡張（エージェント実装＆RAG）**

* **Planner/Researcher/Quizzerの各種エージェントの組み込み。**  
* **Chroma/Qdrant等の軽量ベクターDBのコンテナ追加。**  
  過去の巡礼データからの再探索（RAG）を可能にする。

### **【Phase 5】 召喚コマンド (docker-compose) の完成**

* 開発した全サービスをコンテナ化し、docker-compose up \-d で一発起動できるようにマニフェスト化。

## **6\. Antigravity (AG) 向け命令ガイド**

AGで作業を進める際、以下のルールをプロンプトで繰り返し指定すること：

1. **JSONフォーマットの絶対遵守:** フロント・バック間のデータ授受は常にセクション3の規格を守ること。  
2. **UIデザインはミニマル＆サイバー:** ダークモードを基本とし、ボタンを押しやすいタッチターゲットサイズを確保（しなりの視覚環境への配慮）。  
3. **ローカル環境優先:** バックエンド処理は極力コンテナ内で自己完結させ、LLM処理は外部へのHTTP呼び出し（APIキーは環境変数から取得）で設計すること。
